# Product Requirements Refinement

Use this skill to tighten requirements before technical decomposition.

Repository contract:

- PRDs belong in `docs/specs/PRD-<feature>.md`.
- Follow-on execution specs belong in `docs/work-items/` and are later hydrated via `kanban.publish_specs`.
- Do not rely on GitHub issues as the canonical planning surface for this repository.

## Quality Bar

1. Clarify problem statement and business outcome.
2. Separate must-have and nice-to-have scope.
3. Add measurable acceptance criteria.
4. Identify ambiguous or missing decisions.
5. Capture dependency, rollout, compliance, and UX constraints that will affect refinement-first delivery.
