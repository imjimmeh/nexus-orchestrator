# Probe Result: Memory query provenance / confidence / entity-metadata extension (work item 4f39ed19)

## Narrative Summary

The `memory-query-provenance-extension` scope (kanban work item `4f39ed19-6772-48f3-97f2-8170a3f1d153`, "Extend query_memory to return provenance, confidence, and entity metadata alongside content") is fully implemented across all seven in-scope files and is wired end-to-end through the shared `@nexus/core` schema, the API-side Zod wrapper, the handler projection, the runtime tool descriptor, and the runtime capability contract. The deliverable ships:

- **The three new response fields are present on the shared Zod schema.** `packages/core/src/schemas/memory/query-memory-response.schema.ts` defines `queryMemoryResponseProvenanceSchema` (with `.loose()` passthrough for forward-compat keys), `queryMemoryResponseSegmentSchema` (which exposes `provenance`, `confidence`, plus the entity-metadata fields `entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, and `metadata_json`), `queryMemoryResponseLearningProjectionSchema` (the promoted-lessons block), and the envelope `queryMemoryResponseSchema` that ties them together. The provenance block is nullable so non-`learning_candidate` segments validate cleanly, and `confidence` is bounded `0..1` to match the `RuntimeRecordLearningBody.confidence` contract.

- **The fields are emitted by the handler when `include_provenance: true` (the schema default).** `apps/api/src/workflow/workflow-internal-tools/handlers/memory-tools.handler.ts` resolves `const includeProvenance = params.include_provenance !== false;` (i.e. defaults to `true`) and forwards it into `toQueryMemorySegmentProjection(segment, includeProvenance)`. The helper (`memory-tools.helpers.ts`) calls `synthesizeProvenance(...)` for `learning_candidate` and `fact` source segments by lifting the documented provenance keys (`source_decision_id`, `workflow_run_id`, `job_id`, `agent_profile`, `requested_by`, `scope_type`, `scope_id`, `learning_candidate_id`, `promoted_at`) from `metadata_json`, and copies `confidence` from `metadata_json.confidence` (defaulting to `null` for non-promoted segments). The response then carries entity metadata (`entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, `metadata_json`) on every projected segment alongside the synthesized `provenance` block.

- **The fields are cleanly suppressed when `include_provenance: false` (the QA-fix opt-out path).** When the caller passes `include_provenance: false`, the handler short-circuits `synthesizeProvenance(...)` and sets `provenance: null` for every projected segment, while `confidence` and the entity-metadata fields (`entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, `metadata_json`) continue to be populated so agents can still weight segments by confidence and read attached metadata. The regression test "strips provenance when include_provenance is false" in `memory-tools.handler.spec.ts` pins this contract: it asserts `response.segments[0].provenance === null` while `confidence === 0.87` and `metadata_json.learning_candidate_id` are still present.

- **The schema is shared between `packages/core` and `apps/api/src/workflow/workflow-internal-tools/schemas/memory.ts`.** `apps/api/src/workflow/workflow-internal-tools/schemas/memory.ts` imports `queryMemoryResponseSchema`, `queryMemoryResponseSegmentSchema`, `queryMemoryResponseLearningProjectionSchema`, and `queryMemoryResponseProvenanceSchema` directly from `@nexus/core` (the upstream schema is the single source of truth for the wire format) and extends the per-segment schema with the API-only legacy `metadata` alias via `queryMemoryResponseSegmentSchema.extend({ metadata: ... })`. The shared contract is re-exported from the API wrapper so consumers can import everything they need for the `query_memory` tool from one location, and the runtime capability descriptor `QUERY_MEMORY_RUNTIME_CAPABILITY` in `apps/api/src/workflow/workflow-runtime/workflow-runtime-capability.contracts.ts` exposes the `include_provenance` field in its `bodyMapping` so the runtime contract surface documents it for agent-facing clients. The runtime body schema is `runtimeQueryMemoryBodySchema = queryMemoryBodySchema.extend({ workflow_run_id, job_id, entity_id })`, which preserves the `include_provenance` flag through the HTTP controller hop.

**Confidence: 0.95** — every contractual piece is implemented, the schema is single-sourced through `@nexus/core`, both the default-on and explicit-off `include_provenance` paths are exercised by deterministic unit tests, and the runtime capability surface documents the field for agent-facing clients. The 0.05 deduction reflects the small residual risk that an internal tool consumer (outside `memory-tools.handler.ts`) reads `provenance` without honouring the `include_provenance: false` opt-out — see Open Questions.

## Capability Updates

- **Shared Zod schema** (`packages/core/src/schemas/memory/query-memory-response.schema.ts`): the per-segment shape exposes `provenance` (nullable, `.loose()`-passthrough synthesized block), `confidence` (`number().min(0).max(1).nullable()` matching the `RuntimeRecordLearningBody.confidence` contract), and the entity-metadata fields (`entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, `metadata_json`). The envelope `queryMemoryResponseSchema` adds the promoted-lessons `learning` block (optional, present only when `include_learning: true` is requested).
- **Provenance schema** (`queryMemoryResponseProvenanceSchema`): documented fields are `source_decision_id`, `workflow_run_id`, `job_id`, `agent_profile`, `requested_by`, `scope_type`, `scope_id`, `learning_candidate_id`, `promoted_at`. All fields are `string().trim().min(1).nullable().optional()` except `promoted_at` which is `z.iso.datetime().nullable().optional()`. The schema is `.loose()` so additional provenance keys round-trip without stripping.
- **Handler projection** (`apps/api/src/workflow/workflow-internal-tools/handlers/memory-tools.handler.ts`): `queryMemory(...)` accepts `include_provenance?: boolean` on its params, defaults to `true`, threads the flag into both `projectSegmentsWithUsefulness(...)` (which calls `toQueryMemorySegmentProjection(segment, includeProvenance)` for every segment) and `loadPromotedLearningProjection(...)` (which threads the same flag into the `learning.segments` projection). The handler also accepts the optional `feedback` block (work item 66ea23d1-59f2-451b-a090-a292fad8f21b, milestone 3) and persists one `memory_segment_feedback` row before returning, with a `feedback` acknowledgement on the envelope.
- **Helper projection** (`memory-tools.helpers.ts`): `toQueryMemorySegmentProjection(segment, includeProvenance)` returns `provenance: includeProvenance ? synthesizeProvenance(...) : null` (clean opt-out) while still populating `confidence`, `entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, and `metadata_json`. `synthesizeProvenance(...)` lifts the documented keys from `metadata_json` for `learning_candidate` and `fact` source segments and surfaces `agent_profile_name` under the canonical `agent_profile` key (forward-compat with the documented field name). The shared envelope `apiQueryMemoryResponseSchema.parse(response)` runs on every response (logged via `logger.warn` on failure) so wire-format drift surfaces in production logs.
- **API-side Zod wrapper** (`apps/api/src/workflow/workflow-internal-tools/schemas/memory.ts`): `apiQueryMemorySegmentSchema` extends the shared `queryMemoryResponseSegmentSchema` with the legacy `metadata` alias (backward-compat with the existing handler spec), `apiQueryMemoryResponseSchema` wraps the envelope with the API-only `feedback` acknowledgement block, and the shared `queryMemoryResponseSchema` / segment / projection / provenance schemas are re-exported for one-stop consumers.
- **Runtime tool descriptor** (`apps/api/src/workflow/workflow-internal-tools/tools/memory/query-memory.tool.ts`): `QueryMemoryParams` accepts `include_provenance?: boolean` alongside `include_learning` and `feedback`. `QueryMemoryTool.getDefinition()` returns the `QUERY_MEMORY_RUNTIME_CAPABILITY` from `@nexus/core` so agent-facing clients discover the field via the documented capability contract.
- **Runtime capability contract** (`apps/api/src/workflow/workflow-runtime/workflow-runtime-capability.contracts.ts`): `QUERY_MEMORY_RUNTIME_CAPABILITY` documents `query_memory` with a `bodyMapping` that exposes `include_provenance` (alongside `entity_type`, `entity_id`, `query`, `memory_type`, `include_learning`, `feedback`) and uses the shared `queryMemoryBodySchema` (which declares `include_provenance: z.boolean().optional().default(true)`) as the input schema. The HTTP body schema `runtimeQueryMemoryBodySchema` extends the shared body schema with `workflow_run_id` / `job_id` / `entity_id` for the controller hop.
- **Test coverage** (`packages/core/src/schemas/memory/query-memory-response.schema.spec.ts`): 21 deterministic schema-spec scenarios cover the full promoted-lesson shape (with `provenance` block + `confidence` + `entity_metadata` + `metadata_json`), the preference segment shape (with `provenance: null` + `confidence: null`), the `.loose()` passthrough on provenance, the `confidence` 0..1 range guard, the `memory_type` enum guard, the ISO-8601 timestamp guards, the `metadata_json` free-form record, and the optional / nullable `learning` block.

## Health Findings

- **Schema is single-sourced** through `packages/core` and re-imported by `apps/api/src/workflow/workflow-internal-tools/schemas/memory.ts`. The API-side wrapper extends (not redefines) the shared per-segment schema, so any field added upstream propagates automatically. This matches the `nexus-boundaries/no-core-kanban-residue` and core-share lint policy referenced in `AGENTS.md`.
- **Default-on / opt-out symmetry**: the handler's `const includeProvenance = params.include_provenance !== false;` and the runtime body's `queryMemoryBodySchema` default of `include_provenance: z.boolean().optional().default(true)` are aligned, so a request that omits the field still gets provenance and a request that explicitly sets `false` cleanly suppresses it.
- **Regression test pins the opt-out contract**: `memory-tools.handler.spec.ts` test "strips provenance when include_provenance is false" asserts `response.segments[0].provenance === null` while `confidence === 0.87` and `metadata_json.learning_candidate_id` are still populated. A second regression test in `workflow-runtime-lifecycle.controller.spec.ts` (added in commit `fd3d775be`) wires a real `MemoryToolsHandler` under the mocked `executeInternalTool` to prove the controller forwards the flag verbatim.
- **Defensive defaults for synthesized fields**: `synthesizeProvenance(...)` returns `null` (not an empty object) when no provenance metadata is recoverable so the `provenance` field stays a clean nullable. `readConfidenceFromMetadata(...)` returns `null` (not `undefined`) for non-numeric or missing confidence so the wire-format contract is stable.
- **Validation runs on every response** via `validateQueryMemoryResponse(response, this.logger)` so wire-format drift surfaces in production logs; the failure path is `logger.warn` (not throw) so a malformed projection never breaks a read path.
- **No lint suppression in the assigned files**: none of the seven in-scope files introduce `eslint-disable` / `@ts-ignore` / `@ts-nocheck` comments for the new provenance / confidence / entity-metadata logic.
- **Test style is consistent** with the surrounding handler spec: hand-rolled fake `MemoryManagerService` / `LearningCandidateRepository` / `MemorySegmentFeedbackService`, deterministic clock-free assertions, fixed entity ids (`'project-1'`, `'run-123'`), and BDD-style `describe` / `it` blocks with explanatory comments on the regression sentinels.

## Open Questions

- **Internal tool consumers reading `provenance` directly without honouring `include_provenance: false`**: the handler cleanly suppresses `provenance` when the flag is `false`, but a future internal tool that bypasses `MemoryToolsHandler.queryMemory` and reads `provenance` straight off a `memory_segment` DTO would not honour the opt-out. The current code paths all go through `toQueryMemorySegmentProjection(...)`, so the contract is consistent today; the risk is that a future contributor adds a new query path that reads the DTO directly. **Recommendation**: add a lint guard or contract test that fails if a non-`queryMemory` internal tool reads the `provenance` block without first checking `include_provenance`.
- **`metadata_json` shape is free-form**: the schema documents `metadata_json: z.record(z.string(), z.unknown()).nullable()` so callers can round-trip arbitrary keys, but the provenance-synthesis helper only lifts the documented keys. Additional keys the handler writes (e.g. `tags`, `evidence`, `promotion_policy`) are preserved verbatim on `metadata_json` but never lifted to top-level `provenance`. This is intentional for the current scope; a future "rich provenance" milestone could lift additional keys automatically.
- **`entity_metadata` is a grouping, not a single field**: the work item title uses the umbrella term "entity metadata" to refer to the cluster of fields `entity_type`, `entity_id`, `source`, `created_at`, `last_accessed_at`, `metadata_json`. These are projected as separate first-class fields (not nested under an `entity_metadata` object) so the wire format is consistent with the existing `query_memory` shape and with `memory_segment_feedback` projections. If a future spec requires a nested `entity_metadata` block, the schema change is additive (one new optional object on the segment) and does not break existing callers.
