# Probe Result: Kanban Domain Core (Projects, Work Items, Reviews, Settings)

## Narrative Summary

The Kanban domain core is **fully implemented** as a NestJS service module that owns the project + work-item lifecycle, the review decision path, global Kanban settings, and the board-state aggregation surface that retrospectives and orchestrators depend on. The five scope paths map cleanly to five cohesive subsystems, each registered in `apps/kanban/src/app.module.ts` and exercised by a vitest spec file of its own (the `services/` folder also feeds the `cycle-decision-event.emission.spec.ts` integration test in `services/__tests__/`).

**Projects (`apps/kanban/src/project/`)** — 18 production files plus 10 spec files. `ProjectService` (376 LoC) implements create/list/get/update/delete with full cascade cleanup, plus charter onboarding launch (greenfield/brownfield/refine), repository-workflow settings get/update, blocked-intent reset, and managed-clone orchestration. `ProjectController` exposes 22 endpoints under `/projects/:project_id`, including charter launch + get, memory segments + charter memories CRUD (validated against `PROJECT_MEMORY_CATEGORIES` + `ProjectMemoryCategorySchema`), AGENTS.md GET/PUT with optimistic-locking etag (`ProjectAgentsFileService`), repository branches/files/file-content, and `.nexus/workflows/*.workflow.yaml` read/write/delete via the core client. Charter regen is a full background pipeline: `CharterAggregateService.getCharter()` combines `ProjectGoalsService` + memories into `{ vision, goals, sections }`; `CharterDocRenderService.render()` produces canonical markdown via `CHARTER_SECTIONS`/`CHARTER_SECTION_TO_CATEGORY`; `CharterRegenEnqueuer` schedules debounced BullMQ jobs (2 s delay, 3 attempts, exponential backoff); `CharterRegenProcessor` writes `docs/project-context/CHARTER.md` with `push: true`; `CharterRegenReconciliationService` runs a 15-min sweep (configurable via `KANBAN_CHARTER_RECONCILE_INTERVAL_MS`) over all projects with `basePath`. Managed clone is a hardened git wrapper (`ManagedProjectCloneService`) that validates URL safety (HTTPS-only or SCP-style SSH, no embedded credentials/query/fragment), redacts tokens in audit payloads, gates against `NEXUS_WORKSPACE_BASE_PATH` traversal, integrates with `retrieveSecret` for GitHub tokens, and emits `git.clone.requested|failed|succeeded` events to the core event ledger.

**Work items (`apps/kanban/src/work-item/`)** — 18 production files plus 9 spec files (≈2,800 LoC of tests). `WorkItemService` (≈360 LoC) implements create, list, list-all, paginated query, status transition, dispatch, review-decision, merge, human-feedback resolution, restart, executions lookup, and execution-config get/upsert. Status transitions go through `transitionWorkItemStatus` which executes a `runTransitionGate` "before" hook (blocking) against `executeLifecycleWorkflows` and a non-blocking "after" hook, merging held-state metadata under `metadata.lifecycle.gate` and re-throwing a `ConflictException` with `code: "LIFECYCLE_GATE_BLOCKED"` on blocking failure. Dispatch/review/merge builds a `WorkflowRunRequestV1` with stable `idempotency_key`, `correlation_id`/`causation_id` from the request context, and dynamic Kanban MCP mounts. The `KanbanLifecycleEventPublisher` publishes `kanban.work_item.status_changed.v1` and `kanban.work_item.human_feedback_resolved.v1` to the core domain-event bus with deterministic event ids (sha256 of canonical facts), persisted via `KanbanEventDeliveryProjectionRepository` (pending/accepted/failed), and routed to the orchestration repair lane on delivery failure. Realtime fanout is dual-path: `WorkItemRealtimeGateway` (`@WebSocketGateway({ namespace: "/kanban" })` with a `join-project` subscription) + `WorkItemRealtimePublisher` (Redis pub/sub on `wi:{projectId}` channel, best-effort). The `WorkItemSpecWriter` writes `docs/work-items/{id}.md` markdown with YAML frontmatter (item_id, title, priority, scope, status, depends_on, agent_profile, base_branch, target_branch, context_files) on create, and the controller handles delete cleanup. A `KanbanPermissionsGuard` + `@RequirePermission("work_items:read|write")` decorator is applied at the controller level. The `WorkItemController` is the most extensively routed controller in the project (16 endpoints), with a global `WorkItemGlobalController` providing `/work-items` paginated lookup.

**Reviews (`apps/kanban/src/review/`)** — A thin façade that delegates `recordDecision` to `WorkItemService.submitReviewDecision`; controller exposes `POST /projects/:project_id/reviews/work-items/:workItemId/decision` with `decision: "approve" | "reject"` + `workflow_id` + optional `requested_by`. The `ReviewModule` imports `WorkItemModule` and exports `ReviewService` for Kanban MCP tool consumption.

**Settings (`apps/kanban/src/settings/`)** — Typed, bounded, validated key-value configuration. `KanbanSettingsService` validates keys against `KANBAN_SETTING_DEFAULTS` (12 keys split across two groups: `dispatch` and `auto-restart`), supports `get<T>`, `getNumber` (clamped to `min`/`max`, integer truncation), `getBoolean` (truthy coercion), and `set`; seeds defaults in `onModuleInit` via `seedDefaults`. The `KanbanSettingsController` exposes `GET /kanban-settings` and `PUT /kanban-settings/:key` with `UpdateKanbanSettingRequestSchema` validation. The cooldown default honors `ORCHESTRATION_AUTO_RESTART_COOLDOWN_SECONDS` env var.

**Services (`apps/kanban/src/services/`)** — `BoardStateService` aggregates work-item counts (done/in-progress/blocked/pending), calculates `lastActivityAt`, detects mutations vs. the latest snapshot, and creates `kanban_board_state_snapshots` rows (`BoardStateRepository`) keyed on `(project_id, idempotency_key)` (unique) with a `Like` prefix match. It also has a `detectBoardMutationCore` helper for end-to-end mutation testing. The barrel `services/index.ts` re-exports the service and types. The folder's tests live under `services/__tests__/` (a NestJS project convention) and include a 609-line `cycle-decision-event.emission.spec.ts` integration test that wires `BoardStateService` + `CoreWorkflowClientService` + `CompleteOrchestrationCycleDecisionTool` + `OrchestrationRecordCycleDecisionTool` to verify `kanban.retrospective_cycle_decision_recorded.v1` emission semantics for blocked/complete/repeat-mutated decisions.

**Test coverage across the scope** is comprehensive: 19 dedicated `.spec.ts` files at the service/controller/helper level (covering every public method, every guardrail, every error path), plus 2 repository specs (`kanban-work-item.repository.spec.ts`, `kanban-work-item.repository.query.spec.ts`) covering the work-item persistence layer. Test patterns consistently use in-memory mock repositories that simulate the real `findByProjectAndId`, `replaceDependencies`, `replaceSubtasks`, `save`, `findDependenciesByWorkItemIds`, and `findSubtasksByWorkItemIds` semantics, with realistic date seeding and stable id mapping. The work-item permission decorator is not separately tested in the scope but the controller's metadata is asserted via `Reflect.getMetadata(PATH_METADATA/METHOD_METADATA)`.

## Health Findings

- **Test density**: Every production file in the scope has a companion `.spec.ts` file (30+ spec files covering ~6,000 LoC of test code). The only files without dedicated specs are pure-type barrel modules (`charter-aggregate.service.types.ts`, `charter-regen.queue.ts`, `charter-regen.queue.types.ts`, `work-item-pagination.types.ts`, `work-item-realtime.publisher.types.ts`, `work-item.service.helpers.types.ts`, `work-item.service.types.ts`, `work-item-transition.types.ts`, `work-item.types.ts`, `project-agents-file.service.types.ts`, `project-memory-summary.service.types.ts`, `project-memory-summary.types.ts`, `project.types.ts`, `services/index.ts`, `services/board-state.types.ts`) — these are type-only or constant exports and have no runtime behavior to test.
- **Repository test coverage**: `kanban-work-item.repository.spec.ts` covers `findByExternalSyncRef`, `addTokenSpend`, `linkRunIfUnlinked`, `clearRunLinksIfMatches`, `replaceDependencies`; `kanban-work-item.repository.query.spec.ts` covers `queryWorkItems` filter combinations and ordering.
- **Service spec patterns**: The specs use realistic in-memory mock repositories that simulate TypeORM semantics (createQueryBuilder chaining, save with timestamp refresh, cascade delete). They cover happy paths, every error path (NotFound, BadRequest, Conflict), every guardrail (status, dependency, capacity, branch claim), and every skip reason.
- **Charter regen pipeline coverage**: All three components (`Enqueuer`, `Processor`, `ReconciliationService`) have dedicated specs covering debounce options, BullMQ retry options, push-to-repo behavior, retry on git failure, no-op when basePath missing, and sweep continuation past failing projects.
- **Code quality**: Strict TypeScript, dependency injection throughout, `NotFoundException`/`BadRequestException`/`ConflictException` usage is consistent, `Logger` usage for background/audit operations, error swallowing in fire-and-forget paths (charter regen, real-time publisher) with logged warnings.
- **Settings validation**: `KanbanSettingsService.getNumber` clamps to `min`/`max` and truncates to integer; `getBoolean` coerces "true"/"1"/non-zero to true. Default seeding is idempotent (only seeds missing keys). The `UpdateKanbanSettingRequestSchema` Zod schema validates request bodies at the controller.
- **Lifecycle event correctness**: `KanbanLifecycleEventPublisher` validates inputs via `WorkItemRecordSchema` + `WorkItemStatusSchema` before emit, uses sha256-canonical event ids (deterministic for dedupe), persists delivery state, and routes failures to the orchestration repair lane.
- **No churn signals**: Files in `project/` and `work-item/` were last modified 2026-06-15 alongside the orchestration and dispatch scopes (the recent scope refactor); `review/`, `settings/`, and `services/` are stable from earlier milestones (May–Jun).
- **Settings constants coverage**: The 12 documented settings are spread across `dispatch` (8) and `auto-restart` (3 explicit + 1 env-driven cooldown) groups; spec verifies the numeric bounds-clamping path with a high input being clamped to `max: 50`.
- **Board state schema**: The `kanban_board_state_snapshots` entity has a unique composite index on `(project_id, idempotency_key)` and an index on `project_id`, supporting the `findLatestByProjectId` and `findByProjectIdAndIdempotencyKey` queries.

## Open Questions

The five overlapping `BoardStateSummary` shape declarations enumerated in the revised plan remain a separate type-consolidation follow-up.
- The realtime `WorkItemRealtimeGateway` namespace config derives CORS origin from `process.env.CORS_ORIGIN` but the broadcaster currently only emits `work-item-updated` events; there is no client → server message shape beyond `join-project`. A real client contract was not found in the scope.
- The `charter-regen` BullMQ queue is registered in `ProjectModule` via `BullModule.registerQueue({ name: CHARTER_REGEN_QUEUE })`, but no Redis connection is registered in the module — the queue is presumably shared with the `CoreIntegrationModule` or `ExternalSyncModule` Redis configuration. End-to-end queue wiring was not verified.
- `ProjectService` registers a scope node via `CoreScopeClientService.ensureProjectNode` on every project create; if the core scope client is unavailable the call is logged-and-swallowed (warns) but the project is still persisted — this leaves a "phantom" project in Kanban with no scope node in Core. The reconciliation behavior (if any) for these orphan scope nodes is not visible in the scope.
- The `ManagedProjectCloneService` `assertRepositoryUrlIsSafeRemote` helper accepts SCP-style SSH remotes but the comment notes that credentials cannot be embedded for those, so private SSH repos will fail to clone silently if the user passes them. The user-facing error path is not exercised in the visible specs.
- `WorkItemService.getExecutions()` is a thin pass-through to `coreClient.listWorkflowRuns` and has no scope-level error handling or caching; the actual response shape is not asserted in any spec.
- `WorkItemService.restartExecution()` returns `triggeredRunIds: []` — the "restart" path currently only emits a status-changed event and does not actually re-launch a workflow run. This may be a known gap or may be implemented in a downstream consumer (orchestration continuation). It was not verified end-to-end.
- The kanban settings `KANBAN_SETTING_DEFAULTS` are typed via the contracts package (`KanbanSettingKey`); the 12 keys are not asserted in the controller spec — only the contract envelope and BadRequest on unknown keys are tested. The `description` field is shown as user-facing but the settings controller does not return the `description` for `GET /kanban-settings` in a documented shape beyond the contract's `KanbanSetting` type.
- `ProjectService.resetBlockedIntents` aliases `releaseAllLeases` in the controller; both routes hit the same handler. The `OrchestrationLeaseService.releaseAllForProject` behavior was not visible in the scope and the actual lease semantics (timeout, owner, metadata) are not documented in `resetBlockedIntents`'s surface.
- The `cycle-decision-event.emission.spec.ts` integration test lives under `services/__tests__/` even though it tests MCP tools, the core workflow client, and the retrospective evidence service — this foldering makes `services/` look broader than the production `BoardStateService` it exports. The convention may be by-test-type rather than by-impl-type.
