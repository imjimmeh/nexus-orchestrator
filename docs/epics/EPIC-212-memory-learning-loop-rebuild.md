## Summary

The learning pipeline is **~80% built and structurally sound** (`record_learning` → `learning_candidate` → sweep → `LearningPromotionService` → `memory_segment`; and `SkillProposalService` → `create_skill`). It fails in practice because of **what feeds it**, not how it processes:

1. **Two deterministic templated emitters flood the candidate queue with content-free noise.**
   - `WorkflowPostmortemLearningAggregatorService` (`apps/api/src/workflow/workflow-repair/workflow-failure-postmortem-learning-aggregator.service.ts:213`) emits `"Recurring <failure_class> failures (N occurrences in M days)"` at **0.6 / 0.9** confidence. The `failure_class` is the classifier's **catch-all** (`ambiguous_failure`), so the highest-confidence learnings are the _least_ informative.
   - `WorkflowSuccessLearnerListener` (`apps/api/src/memory/learning/workflow-success-learner.listener.ts:259`) emits `"Workflow run <uuid> for scope <id> completed cleanly in Ns via agent <profile>."` at **0.4**. A success path exists — it is equally content-free.
2. **No run is ever actually read.** Both emitters are counters; neither reads a transcript. The success-transcript read primitive `PiSessionTreeRepository.findRecentSuccessfulForLearning(...)` (added by EPIC-141) has **zero consumers** — it is dead.
3. **Agents cannot easily record what they learn.** The only agent write path is `record_learning` (creates a _candidate_ that dies in the noisy sweep) and `record_strategic_intent` (CEO singleton). There is **no `remember` tool and no prompt directive** instructing agents to capture gotchas/facts/user requests. Operators have never seen an agent create a memory.
4. **The "smart" schema is already built and unpopulated.** `learning_candidates` already declares `score`, `recurrence_count`, `stage_diversity_count`, `failure_reduction_relevance`, `recency_decay`, `source_quality_confidence`, a `signals_json` JSONB column, **and** a ranking index `idx_learning_candidates_status_score` (`apps/api/src/memory/database/entities/learning-candidate.entity.ts:12,45-67`). Nothing writes them. Dedup is exact-SHA256 only, so 26 differently-phrased `ambiguous_failure` candidates survive as 26 rows with `recurrence_count` pinned at 1.

This epic rebuilds the **input and ranking** stages: real agent capture, real success+failure run mining, lightweight ML ranking/dedup, and a governance/lifecycle/measurement layer — converging all sources on the **existing** promotion and skill-materialization pipelines.

### Locked decisions (from product owner, 2026-06-25)

- **Ambition: the full closed loop (Phases 0–3)** including lifecycle and causal measurement, plus the parallel skill-durability workstream.
- **Agent-capture birth path: fast-tracked candidate.** `remember` creates a `learning_candidate` (not a direct `memory_segment`), so every memory shares **one birth path** and inherits scope inference, the _credentials-never-global_ safety rail, and uniform dedup. Explicit user-request captures (`origin: user_request`) are stamped `human_approved_at` and auto-promote near-instantly.
- **Embeddings are in scope; pgvector on the primary DB, dimension-native storage, frontend-managed provider-agnostic model config (updated 2026-06-25, supersedes the original "deferred" decision).**
  - **Image:** primary Postgres `postgres:18-alpine` → `pgvector/pgvector:pg18` (PG major unchanged ⇒ `postgres_data` reused, no dump/restore; org already runs `pgvector/pgvector:pg15` for `honcho-db`, so the image family is proven in-stack). Migration runs `CREATE EXTENSION IF NOT EXISTS vector` (mirroring the `uuid-ossp` pattern at `apps/api/src/database/migrations/20260517000000-api-post-cutover-baseline.ts`).
  - **Dimension-native storage — no fixed-width column, no DDL on model switch.** Embeddings live in a dedicated **`memory_embeddings`** table with an **unbounded `vector` column** (pgvector permits a typeless `vector` holding any dimension), each row tagged `owner_type`/`owner_id`, `model_id`, `dim`, `content_hash`. Retrieval is **exact, scope-filtered cosine KNN** over the **active** model's rows (all share a dim, so `<=>` is valid). Switching a 384-d model for a 1536-d one (or back) is a **data operation** (non-destructive re-embed under the new model, then flip the active pointer, GC the old), never a schema change. _(An unbounded `vector` cannot carry an ANN index — but scope-filtered exact search is fast and more correct than ANN-then-filter at this corpus size; a per-active-model `halfvec` ANN index is a documented scale-only follow-up. A padded fixed-width column was rejected: it doesn't remove the model-tagging requirement, `vector(2048)` forfeits the HNSW index (2000-d cap), and truncation only holds for Matryoshka models — see the Phase-1 plan design note.)_
  - **Config reuses the existing AI-config stack — frontend-managed, provider-agnostic.** Embedding providers/models are ordinary `llm_providers` + `llm_models` rows managed in the existing **/providers** UI, extended with `supports_embedding` + `embedding_dimension` + `default_for_embedding` (mirroring the existing `supports_vision` / `default_for_execution`). Any `provider_id` works (OpenAI, Voyage, Cohere, Google, **custom**); credentials in `secret_store`; **self-hosted / OpenAI-compatible endpoints via `runtime_env.base_url`** — all already supported by the provider model (`llm-provider.entity.ts`). **Anthropic has no embeddings API; the Claude-native path is Voyage (Anthropic's recommended partner).** The active model (and thus the active dimension) is whichever `llm_models` row has `default_for_embedding=true`; changing it triggers a non-destructive re-embed.
  - **No seeded or hardcoded default — embeddings are strictly opt-in.** If **no** active embedding model is configured in the app, **no embeddings are generated** (no external provider is ever contacted by default) and the whole loop runs purely lexical/recency — the Phase-0 behaviour. The existence of a `default_for_embedding` model is the _only_ enable signal; there is no `memory_embedding_enabled` flag and no built-in model constant.
  - **Transport:** `EmbeddingProviderService` resolves the active embedding model via `AiConfigurationService.resolveRunnerProviderConfig(capability:'embedding')`, calls an OpenAI-compatible `/v1/embeddings` (provider adapters for Voyage/Cohere), passes the per-model `dimension` (Matryoshka), and is **fail-soft** (provider down → unembedded → lexical fallback).
  - **Similarity:** `ICandidateSimilarity` primary impl = `EmbeddingSimilarityService` (exact scope-filtered cosine KNN); TF-IDF/MinHash retained as lexical fallback + hybrid arm (genuinely better for exact IDs/paths/commands). Embeddings power **both** candidate dedup/clustering **and** relevance-ranked memory retrieval for injection (replacing today's recency truncation).

## Problem Statement

### The loop is starved and polluted at the input

| Symptom (operator-observed)                                                                      | Root cause                                                                                                            | Evidence                                                                                 |
| ------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- |
| Learning tab shows only templated `"Recurring ambiguous_failure failures (26 occurrences)"` rows | Failure aggregator emits a templated lesson at 0.9 for the classifier's catch-all class                               | `workflow-failure-postmortem-learning-aggregator.service.ts:126-162,213`                 |
| Even "successful-run" learnings are useless                                                      | Success listener emits `"completed cleanly in Ns"` at 0.4                                                             | `workflow-success-learner.listener.ts:166,253-259`                                       |
| No project memories get created                                                                  | The sweep's input is dominated by noise → it rubber-stamps or rejects garbage; nothing of value survives to promotion | `seed/workflows/prompts/memory-learning-sweep/sweep.md`                                  |
| Agents never create memories                                                                     | No `remember` tool; no prompt directive; `record_learning` is a candidate, not a memory, and is rarely granted        | `memory-tools.handler.ts` (no `remember`); base prompt assembly has no capture directive |
| Successful runs are never mined                                                                  | No analyst reads a transcript; the read primitive is dead                                                             | `pi-session-tree.repository.ts findRecentSuccessfulForLearning` (0 consumers)            |
| 26 near-duplicate candidates instead of 1 ranked item                                            | Dedup is exact-SHA256; the scoring/ranking columns are never populated                                                | `record-learning.service.ts createFingerprint`; `learning-candidate.entity.ts:45-67`     |

### Why this is cheaper than it looks

- The promotion pipeline, skill-materialization pipeline, candidate scoring schema (incl. ranking index), success-transcript read primitive, TF-IDF implementation, decay/eviction/drift reapers, usefulness-feedback service, and convergence-metric scaffold **all already exist**. This epic mostly **wires, populates, and replaces** rather than builds greenfield.

## Goals

- An agent can record a durable memory in one tool call, is prompted to do so, and the operator **sees it happen** within a single run.
- Both **successful** and **failed** runs are mined into _real_ lessons/skills (root cause + fix; struggle → working procedure), not counts.
- The two templated emitters are **deleted** (per the aggressive-hygiene mandate); `ambiguous_failure` no longer produces high-confidence noise.
- The candidate stream is **clustered, deduped, and ranked** — the dead scoring columns are populated; the sweep becomes a confirmer of the top-ranked, not a triager of everything.
- Lessons are **routed** to the correct home (project / global / agent-preference / new-skill / skill-patch / drop) with safe, tiered auto-promotion.
- Memory **stays high-signal at volume** (contradiction handling, usefulness-aware decay, drift self-invalidation) and the loop's value is **measurable** (convergence, behaviour-change, holdout lift, cost-per-memory).
- The **right** memories reach the agent: injection moves from blind recency truncation to **embedding relevance-ranked** retrieval (hybrid vector + recency + usefulness, scoped to project + global) within the existing memory token budget.

## Non-Goals

- No second vector store: embeddings live in the **primary** DB (`memory_embeddings`), **not** Honcho/`honcho-db` (which stays the off-by-default chat-memory backend). Single **active** embedding model at a time — runtime-switchable via config + non-destructive re-embed, but **not** per-request multi-model routing. No re-embedding on every edit — only when `content_hash` changes. No semantic search over arbitrary code/transcripts in this epic — only over `memory_segments` and `learning_candidates`.
- No change to the `learning_candidates` external API contract or the `memory_segments` storage model beyond additive JSONB metadata and the migrations named below.
- No Kanban-domain logic in `apps/api`/`packages/core`. The Kanban retrospective producer (`learning.candidate.proposed.v1`) stays on the Kanban side, untouched.
- Not replacing the LLM sweep with a fully deterministic engine — the sweep stays as the promotion confirmer; this epic improves its inputs and ranking.

## Architecture

### Four pillars + one consolidation principle

```
                         ┌─────────────────────────────────────────────┐
   Pillar A (capture) ──▶│  record_learning  →  learning_candidate      │
   Pillar B (mining)  ──▶│  (ONE birth path for all sources)            │
   Kanban retro       ──▶│                                              │
                         └───────────────┬─────────────────────────────┘
                                         ▼
   Pillar C (ML)   EmbeddingSimilarity (pgvector) + lexical fallback + Clusterer
                   + Scorer + TemplateNoiseFilter
                   → populate signals_json + the 5 dead scoring columns + score
                   → embed memory_segments + learning_candidates (memory_embeddings,
                     dimension-native, model-tagged)
                                         ▼
   Pillar D (gov)  LearningRouterService → PromotionGovernancePolicyService
                   → MemoryContradictionService (vector NN) → LearningPromotionService
                                         ▼
                   memory_segment  ──or──  SkillProposalService → create_skill
                                         ▼
                   Loop-1 injection: MemoryRetrievalService (hybrid vector recall,
                   replaces recency truncation) → future agent prompts
                                         ▼
                   Measurement: convergence + behaviour-change + holdout lift
                                         └────────────► feeds back into Pillar C scoring,
                                                        Pillar D governance, and decay/eviction
```

**Consolidation principle (non-negotiable).** The facet analysis independently _triple-built_ the signal/struggle/scoring/similarity layer with conflicting thresholds. To prevent collisions on the shared `learning_candidates` columns and `memory_segments` write path, there is exactly **one** owner of each shared capability:

| Shared capability                                                                             | Single owner                                                                                                                                                 | Module                                                          |
| --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------- |
| Run signal extraction + struggle-then-recovery detection                                      | `RunSignalExtractorService`, `StruggleDetectorService`                                                                                                       | `MemorySignalsModule` (`apps/api/src/memory/signals/`)          |
| Embedding provider/model config (frontend, provider-agnostic, custom + base_url)              | `llm_providers`/`llm_models` (+ `supports_embedding`/`embedding_dimension`/`default_for_embedding`), `ProvidersController`/`ModelsController`, /providers UI | **`AiConfigModule`** (reused, not duplicated)                   |
| Embedding generation (resolve active model via ai-config, batched, async-on-write + backfill) | `EmbeddingProviderService`, `EmbeddingBackfillService`                                                                                                       | `MemorySignalsModule`                                           |
| Non-destructive model/dimension switch (re-embed corpus, flip active, GC old)                 | `EmbeddingReindexService`                                                                                                                                    | `MemorySignalsModule`                                           |
| Hybrid similarity (embedding-primary + lexical fallback) behind `ICandidateSimilarity`        | `EmbeddingSimilarityService` (+ `LexicalSimilarityService`)                                                                                                  | `MemorySignalsModule`                                           |
| Relevance-ranked memory retrieval for injection (hybrid vector + recency + usefulness)        | `MemoryRetrievalService`                                                                                                                                     | `MemorySignalsModule` (consumed by the EPIC-202 injection path) |
| Candidate clustering + fuzzy dedup (embedding-cosine primary, MinHash at scale)               | `CandidateClustererService`                                                                                                                                  | `MemorySignalsModule`                                           |
| Scoring (populate the 5 dead columns + `score`) + template-noise filter                       | `CandidateScoringService`, `TemplateNoiseClassifier`                                                                                                         | `MemorySignalsModule`                                           |
| LLM retrospective orchestration (gate queue, drain, digest, analyst, output router)           | `WorkflowRetrospectiveModule` (`apps/api/src/workflow/workflow-retrospective/`) — **consumes** the above from `MemorySignalsModule`                          | new workflow-adjacent module                                    |
| Routing, governance, contradiction, measurement                                               | learning-module services                                                                                                                                     | `apps/api/src/memory/learning/` + `memory-metrics*`             |

`MemorySignalsModule` owns all reusable detection/similarity/scoring; `WorkflowRetrospectiveModule` owns only the expensive LLM orchestration and depends on the former. **Dedup and clustering work uniformly across agent captures, struggle candidates, and analyst findings because all three are `learning_candidates` scored by the same services.** Both new modules are Kanban-neutral and added to the CLAUDE.md "Workflow Module Boundaries" table.

### Pillar A — In-the-moment agent capture (birth path: fast-tracked candidate)

- **`remember` tool** (`tools/memory/remember.tool.ts`) + `MemoryToolsHandler.remember()`. Agent supplies `content`, optional `memory_type`/`scope`/`tags`/`origin`; **never** entity IDs (resolved from `InternalToolExecutionContext.scopeId`). Routes to `RecordLearningService.recordLearning(...)` creating a candidate with `candidate_type='agent_capture'`, a high `source_quality_confidence` prior, and (for `origin='user_request'`) `human_approved_at = now()` so governance auto-promotes it immediately.
- **`update_memory` tool** — by-id update of an `agent_capture`-origin segment/candidate (version bump; source allowlist prevents editing postmortem/strategic-intent segments).
- **Always-on prompt directive** — a static `memory-capture-guidance` layer in the base agent prompt assembly (`apps/api/src/workflow/workflow-step-execution/step-agent-system-prompt.helpers.ts` base layers — _exact insertion point confirmed in Phase 0_). Tells agents what/when/when-not to record, with the two canonical examples and a "if the user asked, also say you recorded it" line. Suppressed for the sweep/CEO singleton workflows.
- **Default grants at the profile floor** (`seed/agents/*/agent.json` + deny-default workflow `tool_policy`), with a contract test asserting survival of `jobScoped ∩ profileAllowed`.
- **Write-time guardrails** in `MemoryWriteGuardService`: SHA-256 exact dedup (reinforce via `touchReinforcedAt`), near-dup merge via `ICandidateSimilarity` (embedding cosine primary, lexical fallback), per-job write budget, min-length, secret/NUL rejection (reusing `event-ledger` redaction utilities), templated-shape rejection.

### Pillar B — Automated post-run analysis (success + failure)

- **Delete** the two templated emitters' candidate writes. Keep the failure postmortem `memory_segments` write (a useful evidence record) and keep the recurrence **count** — it becomes a _gate signal_, not a candidate.
- **`WorkflowRetrospectiveModule`** with a hybrid trigger: cheap enqueue on terminal events (`WORKFLOW_RUN_COMPLETED_EVENT` / `WORKFLOW_RUN_FAILED_EVENT`) into a `retrospective_queue`, gated by `RetrospectiveGateService` (a cheap deterministic interest score that **inverts** today's pathology — `ambiguous_failure` → low priority; recovered-struggle-on-success → highest). A budget-capped `RetrospectiveDrainScheduler` (BullMQ repeatable) drains top-N; high-signal failures bypass to immediate analysis.
- **`RunTranscriptDigestService`** compresses `pi_session_trees.jsonl_data` to a token-bounded, struggle-anchored digest (`TokenCounterService.countJSONLTokens`). Single biggest cost lever. **Built against real samples (`retrieve-session-logs`) in Phase 0 spike** — fail-soft to an event-ledger-only digest.
- **Retrospective Analyst** (`seed/workflows/run-retrospective.workflow.yaml` + `retrospective-analyst` profile, light tier, read-only tools) reads the digest and emits structured `findings[]` (memory / skill_proposal / none), each citing `evidence_event_ids`. Prompt demands generalizable diagnosis (root cause + fix; working procedure), not narration; returns `none` rather than inventing.
- **`RetrospectiveOutputRouter`** routes findings into the **existing** `record_learning → sweep → promote` and `create_skill_proposal → create_skill` pipelines. Confidence is **re-derived** by the router (struggle-backed → up to ~0.7; pure inference → capped ~0.45) so a hallucinated 0.95 cannot auto-promote.

### Pillar C — Lightweight ML / signal layer + embedding similarity (pgvector)

- **pgvector foundation** — primary DB image → `pgvector/pgvector:pg18` (volume-compatible, see Locked Decisions); `CREATE EXTENSION vector`; dimension-native **`memory_embeddings`** table (unbounded `vector`, tagged `owner_type`/`owner_id`/`model_id`/`dim`/`content_hash`) serving exact scope-filtered cosine KNN — **no** fixed-width column, so model/dimension switches need no DDL. Test stacks (`packages/e2e-tests/src/stack/containers.ts`, integration specs that run `migrationsRun: true`) move to the same image so the extension migration applies. Full task breakdown + the dimension-native design note in the [Phase-1 pgvector/embeddings plan](../superpowers/plans/2026-06-25-memory-learning-loop-phase-1-pgvector-embeddings.md).
- **Embedding model config (frontend, provider-agnostic, opt-in)** — reuses the existing AI-config stack: `llm_providers` + `llm_models` rows in the **/providers** UI, extended with `supports_embedding` + `embedding_dimension` + `default_for_embedding`. OpenAI / Voyage / Cohere / Google / **custom** providers; self-hosted via `runtime_env.base_url`; credentials in `secret_store`. The active model = `default_for_embedding`; switching it triggers a non-destructive re-embed (`EmbeddingReindexService`). **No seed/hardcoded default**: with no model configured, nothing is embedded and the loop falls back to lexical/recency (the existence of an active model is the sole enable signal).
- **`EmbeddingProviderService`** — resolves the active embedding model + credential via `AiConfigurationService.resolveRunnerProviderConfig(capability:'embedding')`; OpenAI-compatible `/v1/embeddings` transport (Voyage/Cohere adapters); passes the per-model `dimension` (Matryoshka). Batched, retried, **fail-soft** (provider down → row stays unembedded, lexical fallback covers it). Embeds on write via a BullMQ job (write path never blocks on a network call); **`EmbeddingBackfillService`** walks existing rows in batches.
- **`TemplateNoiseClassifier`** — regex on the two known templates + low-signal heuristics (no file/table/tool/command/credential anchor). Floors `source_quality_confidence=0.2`, excludes from the sweep queue. Cheapest, ~80% noise reduction on day one.
- **`RunSignalExtractorService` + `StruggleDetectorService`** — cheap signals (retry count, tool-error-then-success, repeated-failed-command, novelty via tool-sequence n-grams, duration/token outliers) feeding the interest gate and carrying real evidence onto candidates.
- **`EmbeddingSimilarityService`** (primary impl of `ICandidateSimilarity`) — pgvector cosine KNN for near-dup detection; **`LexicalSimilarityService`** (the extracted TF-IDF/MinHash `tokenize()`) is the fallback when a row is unembedded/provider-down and the lexical arm of a hybrid score (reciprocal-rank fusion) that recovers exact IDs/paths/commands embeddings blur.
- **`CandidateClustererService`** — embedding-cosine agglomerative clustering (threshold) over pending candidates as primary; MinHash/LSH retained for scale fallback. Canonical elected; `recurrence_count` set to cluster size; rest marked `merged` (recoverable via `diagnostics_json.merged_into`). The 26× `ambiguous_failure` collapse to one row with `recurrence_count: 26` on **semantic** similarity, not just lexical.
- **`MemoryRetrievalService`** — the read side: embeds the current step/task context, hybrid-ranks `memory_segments` (vector cosine × recency × usefulness × pinned) scoped to project + global, and returns top-K within the memory token budget. Replaces the recency-ordered truncation in the EPIC-202 injection path (`memory-segment.repository.ts` ordered reads). Directly answers the operator ask to _"add embeddings retrieval."_
- **`CandidateScoringService`** — populates `recurrence_count`, `stage_diversity_count`, `recency_decay` (`exp(-λΔdays)`), `source_quality_confidence` (per-source prior; templated = 0.2), and composite `score` (logistic, hand-set β, weights in `system_settings`). Uses the existing `idx_learning_candidates_status_score`. A scheduled signal-scoring pass runs before the 2am sweep.
- **Sweep input contract** — `list_pending_learning_candidates` returns clustered, score-ranked, template-filtered candidates with `signals_json`; `sweep.md` updated to consume top-ranked-to-budget and trust `struggle_backed` / `agent_capture` provenance.
- _(Phase 3, nice-to-have)_ **`FeedbackWeightTunerService`** — weekly L2 logistic regression over promoted+usefulness labels → bounded, versioned weight updates (`signal_weight_history`).

### Pillar D — Quality, governance, lifecycle, measurement

- **`LearningRouterService`** — deterministic scope inference (scope-diversity, agent-concentration, lesson-shape regex, procedural length, existing-skill similarity, code-reference presence) → routing target; LLM arbitration only on ties / low scope-confidence. **Rules:** credential/connection facts → project, pinned, **never global**, never the secret value; reusable procedure → `skill_new`; refinement of existing skill → `skill_patch`; cross-scope truth (≥3 scopes) → global; single-scope → project; behavioural "always/never" concentrated on one profile → `agent_preference`. Durable `routing_target` column.
- **`PromotionGovernancePolicyService`** — tiered matrix: project facts auto-promote (provisional) at high confidence; agent-preference stricter (≥0.8); **global never auto**; skills always proposal; templates auto-drop. Auto-promoted items carry `governance_state='provisional'` + probation.
- **Redesigned Learning tab** — `LearningTabClusterCard.tsx` (clusters, scores, routing badge), `LearningTabDiffPreview.tsx` (before/after for skill-patch + memory contradiction/update), score-breakdown popover (the now-populated axes), "why this scope" rationale + source-run links, suppressed-noise drawer ("127 low-signal items auto-filtered").
- **`MemoryContradictionService`** — vector nearest-neighbour (lexical fallback) + heuristic stance check (LLM confirm on hit) → supersede / version / archive (`supersedes`/`superseded_by`); never silently keep both; ambiguous → human diff.
- **Lifecycle tuning** — replace the blanket `MEMORY_DECAY_EXEMPT_SOURCES` with a value-based predicate (`pinned OR usefulness ≥ θ OR injected_and_helped`); join `MemorySegmentFeedbackService.usefulness_ratio` into `findDecayCandidates`/`findEvictionCandidates` (currently collected-but-unused); attach drift references to code-anchored facts so `MemoryDriftDetectionService` self-invalidates stale memories. Ship behind a flag in shadow mode for one window first.
- **Measurement** — verify/wire the terminal-outcome observer first (`convergence_ratio` numerator); add a behaviour-change counter (did the injected lesson's anchored tool/path get used post-injection?), an A/B holdout `nexus_learning_lift{scope}`, cost-per-promoted-memory, and a provisional-memory probation evaluator in `MemoryMetricsRefreshService`. A "Learning Health" web panel rolls these up.

| Boundary lint (`nexus-boundaries/no-core-kanban-residue`)                                                    | Low                  | All new modules Kanban-neutral by construction; `signals_json` must carry no Kanban identifiers                                                                                                                                                                                                      |
