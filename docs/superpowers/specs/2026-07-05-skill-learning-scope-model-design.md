# Skill/Learning Scope Model — Design

Date: 2026-07-05
Status: Draft, approved for planning

## Problem Statement

Skill proposals generated by the self-improvement/retrospective pipeline
(`WorkflowRetrospectiveModule`) land as effectively **global** — visible to
every agent profile in every project — even though most learnings originate
from, and are only relevant to, a single project. The user wants three tiers
of applicability:

1. **Global** — available everywhere.
2. **Project** — available to any agent operating within a specific scope.
3. **Project + agent** — available only to a specific agent profile within a
   specific scope.

This is not "add scoping from scratch." Investigation found a scope model
already exists in multiple places, partially wired, with specific gaps that
explain the observed all-global behavior.

## Current State (as of this investigation)

- `AgentSkill`/`AgentProfileSkill` (`apps/api/src/ai-config/database/entities/`)
  — the DB rows backing skill assignment — carry **no scope column**.
  `AgentSkill` is unique only on `name`.
- The actual runtime skill _content_ lives file-based
  (`AgentSkillLibraryService`, `storage/skills/<name>/SKILL.md`), which
  **does** support a `scope: { projects[], agents[], workflows[] }`
  frontmatter block, resolved via `listSkillsForScope()` — matched with OR
  semantics across the three independent lists (no way to require "project Y
  **and** agent X").
- That frontmatter scope **is** wired into both runtime consumers
  (`WorkflowStageSkillPolicyService.resolveAssignedSkills`, called from both
  the step-execution path (`step-support.service.ts`) and the
  subagent-provisioning path (`subagent-orchestrator.skills.helpers.ts`)) —
  this part is not dead code.
- But it is only ever **populated** for brand-new skills
  (`skill_create` proposals), by `SkillCreateCompletionListener.tryAutoApplyScope`,
  and only when the `skill_scope_confirmation_mode` system setting
  (`apps/api/src/settings/skill-scope-confirmation.settings.constants.ts`) is
  `'auto'`. The default is `'manual'`, and **no confirm endpoint or UI exists
  anywhere in the codebase** — a `'manual'`-mode proposal parks
  `provenance.materialization.scope_confirmation.pending: true` forever.
- `skill_assignment` proposals (bind an _already-existing_ skill to a
  target — the router's default whenever `AgentSkillsService.skillExists()`
  is true, i.e. the common case per `RetrospectiveOutputRouter.routeSkillProposal`)
  **never touch scope at all**. `SkillAssignmentApplier` →
  `applySkillAssignments` → `AgentSkillsService.addProfileSkillsByProfileName`
  pushes onto `agent_profiles.assigned_skills`, a flat `string[]` column with
  no room for scope metadata.
- The originating `scope_id` **is** captured correctly at proposal-creation
  time (`RouteContext.scopeId` → `provenance.scope_id`,
  `retrospective-output-router.service.ts:349`) — a hard fact about where the
  learning came from — but it is only forwarded as a plain input to the
  `create_skill` workflow run and never reaches either the skill's frontmatter
  scope or any assignment record.
- Separately, the codebase has a real, implemented multi-tenant scope tree:
  `scope_nodes` (`apps/api/src/scope/database/entities/scope-node.entity.ts`;
  `parentId`, `type: platform|org|region|team|project`, `isTenantRoot`) with a
  `scope_node_closure` ancestor/descendant table, and `role_assignments`
  (`userId`, `roleId`, `scopeNodeId`, unique together) checked via
  `AuthorizationService.can(userId, permissionName, scopeNodeId)`
  (`apps/api/src/auth/authorization/authorization.service.ts`), which already
  does ancestor-inclusive permission resolution through the closure table. A
  `skills` resource with `skills:read/create/update/delete/manage` permissions
  and a seeded `skills_admin` role already exist
  (`apps/api/src/auth/authorization/permission-catalog.ts`).
- `agent_profiles.scope_node_id` and the separate `skills` DB corpus table's
  `scope_node_id` both already anchor on `scope_nodes`, via
  `ScopedConfigResolver`, for a different purpose (LLM/provider config
  resolution) — confirming `scope_nodes` is the one real generic scope
  entity in this codebase, not a second thing to invent.

## Goals

- Pipeline-created skills and skill assignments default to the scope of the
  run they came from, not global.
- Support three tiers: global, project (`scope_node_id`), and project+agent
  (`scope_node_id` + `agent_profile_id`).
- An autonomous proposal can never grant itself scope wider than the scope it
  actually ran under. Widening (including to global) always requires a human
  with a real permission grant at the _target_ scope node — reusing the
  existing `scope_nodes`/`role_assignments`/`AuthorizationService` machinery,
  not a bespoke authorization concept.
- Fully additive: no change to how a human operator directly assigns a
  global skill via the existing admin UI/API.

## Non-Goals

- Rebuilding or extending the `scope_nodes` tenancy tree itself — it already
  exists and already does ancestor-aware permission resolution.
- Retrofitting the `workflows[]` frontmatter scope dimension — untouched,
  orthogonal to this work.
- A staged/bulk confirmation UI beyond the single confirm/reject action
  described below (`'staged'` mode's future bulk-confirm pass stays out of
  scope, unchanged).

## Design

### 1. Scope anchor: `scope_nodes`, not a bespoke `scope_id` string

`scope_id`, wherever it appears in the improvement/retrospective pipeline
(`ctx.scopeId`, `provenance.scope_id`, `resolveScopeId`), is a
`scope_nodes.id`. Any consumer that receives a `scope_id` string must
validate it resolves to a real, non-archived `scope_nodes` row before acting
on it — if it doesn't (e.g. a stale or malformed value on an older trigger
payload), treat the proposal as scope-unknown and fall back to the existing
`'manual'`-pending behavior rather than guessing or defaulting to global.

### 2. New table: `agent_profile_skill_bindings`

Mirrors the existing `workflow_skill_bindings` pattern exactly (additive,
runtime-only, separate from the flat `assigned_skills` column so it can never
be clobbered by a profile reseed):

```
agent_profile_skill_bindings
  id               uuid primary key
  agent_profile_id uuid null   -- null = applies to any profile within scope_node_id
  scope_node_id    uuid not null references scope_nodes(id)
  skill_name       varchar(64) not null
  provenance       jsonb null
  created_at, updated_at
```

Unique on `(COALESCE(agent_profile_id,''), scope_node_id, skill_name)` via an
expression index, same technique `workflow_skill_bindings` already uses for
its `COALESCE(step_id, '')` uniqueness.

Two row shapes cover the two new tiers:

- `agent_profile_id: null` → **project tier**: any agent profile operating
  under `scope_node_id` gets this skill.
- `agent_profile_id: <id>` → **project+agent tier**: only that profile, only
  under that scope.

The existing flat `assigned_skills` array on `agent_profiles` is untouched
and remains the **global tier** — a skill assigned there applies regardless
of scope. This table is never used to represent global assignment, so there
is exactly one representation per tier, no overlap to reconcile.

### 3. Resolution logic changes

`WorkflowStageSkillPolicyService.resolveBaseSkillSet` (currently: global
profile skills ∪ frontmatter-`listSkillsForScope`-matched skills, scoped
wins on name collision) gains a third source: query
`agent_profile_skill_bindings` for rows where
`(agent_profile_id IS NULL OR = current agentProfile)` and
`scope_node_id` is in the ancestor-closure set of the run's current
`scope_id`... **decision needed at plan time**: should a project-tier or
project+agent-tier binding at an _ancestor_ scope (e.g. an org-level binding)
apply to a descendant project, mirroring how `AuthorizationService` already
walks the closure table? Recommended default: yes, reuse
`scope_node_closure` for this lookup too, for consistency with every other
scope-aware read path in the codebase — but flag this explicitly in the
implementation plan since it changes blast radius (an org-wide skill binding
would silently reach every project under it).

Because binding rows and the flat global array can never name-collide in a
way that changes _content_ (same `skill_name` → same skill content either
way), there is no real precedence conflict to resolve — multiple sources
naming the same skill simply dedupe, same as today's `resolveBaseSkillSet`
comment already describes for the frontmatter case.

### 4. Origin scope vs. recommended scope

Split what happens at skill materialization/assignment time into two
distinct actions:

- **Origin scope** (`provenance.scope_id`, a fact, not a judgment call) is
  applied **immediately and unconditionally**, regardless of
  `skill_scope_confirmation_mode`:
  - `skill_create`: written to the new skill's frontmatter `scope.projects`
    (existing mechanism, just now always populated instead of gated).
  - `skill_assignment` / post-materialization `assignment_targets` with an
    `agent_profile` target: instead of the current unconditional
    `addProfileSkillsByProfileName` call, insert an
    `agent_profile_skill_bindings` row `{ agent_profile_id: <resolved
profile>, scope_node_id: provenance.scope_id, skill_name }`. If
    `provenance.scope_id` is absent or fails to resolve to a real
    `scope_nodes` row (e.g. a manually-triggered proposal with no run
    context), fall back to today's global flat-array assignment — there is
    no narrower scope to default to when none is known, and manual
    proposals are, by construction, an intentional human action already.
- **Recommended scope** (the LLM's `recommended_scope` output, which may
  suggest something broader than or different from the origin — e.g.
  "generalize to all projects") stays gated behind
  `skill_scope_confirmation_mode` **and additionally** requires a widening
  check (below) regardless of mode. `'auto'` mode now means "auto-confirm a
  recommendation that is equal to or narrower than the origin scope";
  anything that would widen past the origin scope is never auto-applied by
  any mode — it always requires the confirm action.

### 5. Escalation guard (widening confirmation)

Add a confirm action (currently missing entirely — `'manual'` mode has no
way to ever leave the pending state today) that:

1. Resolves the _target_ scope node the recommendation would widen into
   (e.g. `GLOBAL_SCOPE_NODE_ID` for a "make this global" recommendation, or
   a specific other `scope_nodes.id`).
2. Requires the confirming user to hold `skills:update` (reuse the existing
   `skills` permission resource — no new permission needed) at that target
   scope node or an ancestor of it, checked via the existing
   `AuthorizationService.can(userId, 'skills:update', targetScopeNodeId)`
   (already ancestor-inclusive via `scope_node_closure`).
3. If nobody holds that permission, the recommendation simply cannot be
   confirmed and stays parked at the already-applied origin scope
   indefinitely — this is correct, expected behavior (absence of an
   authorized approver means "never widens"), not an error state to special-
   case.
4. On confirm, apply the recommended scope the same way
   `tryAutoApplyScope`/`buildScopedMarkdown` do today. On reject, clear
   `scope_confirmation.pending` without changing the already-applied origin
   scope.

This is a REST endpoint on the existing improvement-proposals API
(`apps/api/src/improvement/`), not a new module — plan-time detail.

### 6. Migration

Per explicit instruction: wipe existing state rather than reconcile it.

- Truncate/clear all `agent_profiles.assigned_skills` entries (all of them,
  including hand-configured ones — explicit user decision, not limited to
  pipeline-created rows).
- Delete all `AgentSkill`/`skills`-corpus rows and their on-disk
  `storage/skills/<name>/` directories.
- Delete all `skill_create`/`skill_assignment` `improvement_proposals` rows
  (or reset them to a clean terminal state — plan-time detail depending on
  whether proposal history should be preserved for audit purposes elsewhere).

### 7. Testing

- Unit: `agent_profile_skill_bindings` repository/service (mirrors existing
  `WorkflowSkillBindingService` test patterns).
- Unit: `resolveBaseSkillSet` (or its successor) with global/project/
  project+agent combinations, including the ancestor-closure widening case.
- Unit: `routeSkillProposal` / appliers — origin-scope auto-apply, widening
  requires confirm, missing/invalid `scope_id` falls back to global-manual
  behavior.
- Unit: confirm endpoint — permission-at-target-scope enforcement, both
  allow and deny paths, using the existing `AuthorizationService` test
  doubles.
- Integration: end-to-end retrospective → proposal → applier → effective-
  skills resolution for a project-scoped and a project+agent-scoped case,
  confirming a different project's agent does _not_ see the skill.

## Open Questions (for the implementation plan, not blocking spec approval)

- Should project/project+agent bindings inherit down the `scope_nodes` tree
  by default (Section 3), or only apply at the exact scope node? Recommended:
  inherit, for consistency with every other scope-aware path in this
  codebase, but confirm before implementation since it changes who sees a
  skill.
- Should `skill_create`/`skill_assignment` proposal rows be preserved
  (soft-reset) or hard-deleted during migration? Either is consistent with
  "wipe existing state"; pick based on whether proposal audit history has
  value elsewhere.
