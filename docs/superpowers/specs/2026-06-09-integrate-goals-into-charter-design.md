# Design: Integrate Goals into the Charter

**Date:** 2026-06-09
**Status:** Approved (design), pre-implementation
**Follows:** EPIC-203 (Conversational Project Onboarding & Charter), `2026-06-08-project-intent-tab-design.md`
**Theme:** Make project goals a first-class *section of the charter* — unified in data, API, prompts, and UI — instead of a separate, parallel artifact.

---

## 1. Problem

EPIC-203 shipped a charter that is conceptually integrated with goals but structurally separate. Today there are **three disconnected stores**, presented as two disconnected UI columns:

| Artifact | Storage | Written by | Read by web via |
|---|---|---|---|
| **Goals** | `kanban_project_goals` table (status, MoSCoW, priority, worklogs, work-item links) | `kanban.goal_*` MCP tools / REST `/goals` | `useProjectGoals` → `GoalsTab` (left column) |
| **Charter memories** | `memory_segments` (`entity_type='project'`, `metadata.category`) — 9 categories | `record_project_memory` tool / REST `/charter-memories` | `useCharterMemories` → `CharterColumn` (right rail) |
| **CHARTER.md** | markdown file in the project git workspace, 10 sections incl. a `## Goals` | `update_charter` runner tool | *nothing* — never served to the web |

Consequences of the split:

- **Triple-storage of "goals":** structured board rows, a `## Goals` section in `CHARTER.md`, and (potentially) overlapping narrative — with **no defined synchronization**. The onboarding prompts leave the boundary ambiguous.
- **`CHARTER.md` is write-only:** generated by the agent but never read back by the web app; its only consumer is the discovery prompt.
- **Section/category vocabularies don't line up:** `CHARTER_SECTIONS` has *Vision* and *Success Criteria* (no memory category, so they live only as file prose), while memory categories have *decision* and *preference* (no canonical charter section).
- **The UI is two columns with no shared data, hooks, endpoints, or types** — the separation the user dislikes is baked in at every layer.

## 2. Goal

One unified **Project Charter** surface where goals are a structured *section* alongside categorized intent. The **DB is the single source of truth**; `CHARTER.md` is a generated, always-fresh export for git + the discovery phase. Goals keep all their first-class power (status, MoSCoW, priority, worklogs, work-item links).

### Approved decisions (from brainstorming)

1. **Integration depth:** *Charter as the home, goals stay structured.* Goals remain first-class entities but become a formal section of one charter surface.
2. **Source of truth:** *DB is truth, `CHARTER.md` generated.* The duplication is killed at the root — the file's `## Goals` section is auto-rendered from board goals.
3. **Narrative home:** *New memory categories.* Add `vision` and `success_criteria` so the whole charter = board goals + categorized memories, one uniform mechanism.
4. **UI:** *Single sectioned document.* One scrolling "Project Charter" ordered by canonical sections; Goals is one section rendering the existing rich goal cards inline.
5. **Freshness:** *Always-fresh `CHARTER.md`*, regenerated event-driven on any charter mutation.

### Prerequisite: charter tooling lives in kanban, not api

Charter and project intent belong to **projects**, which are a **kanban** concern. The current placement of `update_charter` and `record_project_memory` as **`apps/api`** workflow-internal-tools is architecturally wrong. Before this integration work, those tools — plus other misplaced project/kanban-scoped agent tooling (e.g. the `propose_work_items` tool) — are relocated from `apps/api` to `apps/kanban` as MCP tools, tracked by a **separate relocation plan** (`docs/superpowers/plans/2026-06-09-relocate-kanban-tooling-api-to-kanban.md`). This design assumes that move has landed: every charter tool referenced below is kanban-side, so all charter writes originate in kanban and the event-driven regeneration is fully in-process (no cross-service signalling).

## 3. Architecture

```
                    ┌─────────────────────── apps/kanban ───────────────────────┐
  Web edits ──REST──▶ ProjectGoalsService (goals)                                │
  Agent (MCP) ──────▶ ProjectMemorySummaryService (charter memories)            │
                    │        │ (mutation)                                        │
                    │        ▼ enqueue (debounced, jobId=charter-regen:<proj>)   │
                    │   charter-regen BullMQ queue                               │
                    │        │                                                   │
                    │        ▼                                                   │
                    │   CharterDocRenderService ── reads goals + memories        │
                    │        │  renders markdown (canonical section order)       │
                    │        ▼  coreClient.writeRepoFile(base_path, CHARTER.md)  │
                    └────────┼──────────────────────────────────────────────────┘
                             ▼  (HTTP, existing)
                    apps/api GitController POST /git/files/write → write + AUTO-COMMIT
                             ▼
                    docs/project-context/CHARTER.md  ← read by discovery (always fresh)

  Web read: GET /projects/:id/charter  → aggregate { vision, goals[], sections{...} } → useCharter → CharterDocument
```

Key property: **no runner/container is involved in file generation.** `coreClient.writeRepoFile()` already exists in kanban and performs write+commit in one call against the project's `base_path` checkout.

## 4. Detailed Design

### 4.1 Contracts — `packages/kanban-contracts/src/project-charter.schema.ts`

- Extend `PROJECT_MEMORY_CATEGORIES` with **`vision`** and **`success_criteria`**.
- Reconcile sections ↔ categories so every memory category maps 1:1 to a charter section, plus the single non-memory **Goals** section. New canonical `CHARTER_SECTIONS` order:

  `Vision · Goals · Requirements · Constraints · Dos & Don'ts · Non-Goals · Success Criteria · Decisions · Preferences · Glossary · Stakeholders · Open Questions`

- Export a single **`CHARTER_SECTION_TO_CATEGORY`** map (the one source of truth used by the renderer, the aggregate endpoint, and the web UI ordering — DRY).
- No DB migration: categories are `metadata.category` strings.

### 4.2 Charter document renderer — `apps/kanban` (new `CharterDocRenderService`)

- Pure function-ish service: given `projectId`, reads board goals (`ProjectGoalsService.listGoals`) + charter memories (`ProjectMemorySummaryService.getCharterMemories`), renders the full `CHARTER.md` in canonical section order.
- `## Goals` is rendered from board goals (title, status, MoSCoW, priority, description) — never hand-authored.
- Idempotent, full-file render (mirrors the old tool's `CHARTER_PATH_SEGMENT` and section vocabulary; reuses the contracts map).

### 4.3 Always-fresh regeneration (event-driven, debounced)

- New BullMQ queue **`charter-regen`**. Enqueue with a deterministic `jobId = charter-regen:<projectId>` and a short trailing `delay`, so a burst of edits collapses into one job (pattern proven in `external-sync-polling.scheduler.ts` / `project-orchestration-wakeup.service.ts`).
- **Triggers (service layer, so REST *and* MCP paths both fire):**
  - `ProjectGoalsService` mutations (`createGoal`, `updateGoal`, `updateStatus`, `setArchived`, `reorderGoals`, `createWorklog`/`linkWorkItem` are optional — title/status-affecting ones at minimum).
  - `ProjectMemorySummaryService` charter-memory mutations (`createCharterMemory`, `updateCharterMemory`, `deleteCharterMemory`).
  - `record_project_memory` (a kanban MCP tool after the relocation prerequisite): writes project memories through kanban's memory write path, which enqueues the same debounced regen **in-process** — no cross-service signal needed. This is a direct payoff of the relocation: every charter write now originates inside kanban.
- **Processor:** render via `CharterDocRenderService`, then `coreClient.writeRepoFile({ repoPath: project.basePath, filePath: 'docs/project-context/CHARTER.md', content, message: 'docs(charter): regenerate from project intent' })` (write + auto-commit). Tolerate projects without a `base_path` (skip quietly).

### 4.4 Unified read API — `apps/kanban` `ProjectController`

- New **`GET /projects/:id/charter`** → `{ success, data }` where `data = { vision: CharterMemoryItem|null, goals: ProjectGoal[], sections: Record<category, CharterMemoryItem[]> }`, composed from `ProjectGoalsService` + `ProjectMemorySummaryService`, ordered by the contracts map.
- Existing `/goals/*` and `/charter-memories/*` **write** endpoints stay (the aggregate is read-only). Fix the `entity_type` inconsistency (`'project'` vs `'kanban.project'`) so all charter views read the same rows.

### 4.5 Remove `update_charter` (post-relocation, in `apps/kanban`)

- **Delete** the relocated `update_charter` tool (moved into `apps/kanban` by the relocation plan) and its spec — eliminate, don't deprecate. The agent no longer writes the file at all; the file is fully derived.
- Remove `update_charter` from CEO `agent.json` tool policy and the authority-contract test. No replacement runner tool is needed — regeneration is kanban-owned and automatic.

### 4.6 Web UI — `apps/web` single sectioned charter

- Replace `ProjectIntentTab`'s two-column layout with one scrolling **`CharterDocument`** driven by a single `useCharter(projectId)` hook hitting `GET /projects/:id/charter`.
- **Reuse, don't rewrite:** the Goals section embeds the existing goal components from `GoalsTab.tsx` (status/MoSCoW/worklogs/work-item links intact); other sections reuse `CharterCategorySection`; Vision renders as a single-item section. Keep optimistic create/update/delete against the typed write endpoints.
- Fix the legacy tab `value="goals"` → `value="charter"` (label already "Project Intent").
- Retire `CharterColumn.tsx` (its category-section rendering migrates into `CharterDocument`).

### 4.7 Prompts / workflow / agent — `seed/`

- `onboard.md`, `brownfield-onboard.md`, `refine.md`, CEO `PROMPT.md`: drop the three-way split and the "Narrative → `update_charter`" instruction. New loop is two stores: **goals → `kanban.goal_*`; everything else (incl. Vision, Success Criteria, Decisions, Preferences) → `record_project_memory` with a category.** The `CHARTER.md` file is no longer the agent's concern — it materializes automatically. Update dedup guidance in `refine.md` accordingly.
- `discovery.md`: unchanged in intent — still reads `docs/project-context/CHARTER.md` (now always fresh) + `query_memory` as ground truth. Remove any assumption that the agent must (re)generate the file.
- CEO `agent.json`: remove `update_charter`; goal tools + `record_project_memory` remain. Update `ceo-authority-contract.test-helper.ts`.

## 5. Components & Boundaries

| Unit | Responsibility | Depends on |
|---|---|---|
| `project-charter.schema.ts` (contracts) | Canonical categories, sections, section↔category map | — |
| `CharterDocRenderService` (kanban) | Render `CHARTER.md` markdown from goals + memories | goals service, memory summary service, contracts |
| `charter-regen` queue + processor (kanban) | Debounced regen → write+commit | render service, `coreClient.writeRepoFile` |
| Regen triggers (kanban services + lifecycle consumer) | Enqueue on any charter mutation | the queue |
| `GET /projects/:id/charter` (kanban) | Read-only aggregate for the web | goals + memory services, contracts |
| `CharterDocument` + `useCharter` (web) | Single sectioned charter UI | aggregate endpoint, existing goal/memory write hooks |
| Prompts/agent (seed) | Capture intent into goals + memories only | goal + memory tools |

## 6. Testing

- **Contracts:** new categories present; section↔category map total and 1:1 for memory categories.
- **Renderer:** Goals section rendered from board goals (not memory); sections ordered canonically; empty sections handled; path is fixed `docs/project-context/CHARTER.md`.
- **Regen:** debounce coalesces a burst into one job (stable `jobId`); processor calls `writeRepoFile` with `base_path`; missing `base_path` skipped; lifecycle-stream `charter.changed` enqueues.
- **Aggregate endpoint:** returns vision + goals + sections in order; `{success,data}` envelope; `entity_type` consistency.
- **Removal:** `update_charter` gone from registration + agent policy; authority-contract test denies it; `validate:seed-data` passes.
- **Web:** `CharterDocument` renders sections in order; goal interactions still work; optimistic memory edits + rollback.
- **Workflow seed contract:** `project-charter-ceo` jobs still valid after prompt edits.

## 7. Risks & Trade-offs

| Risk | Mitigation |
|---|---|
| Regen storm on rapid edits | Debounced BullMQ job with per-project `jobId` collapses bursts into one commit |
| Commit noise in project git history | Single coalesced commit per quiet window; clear `docs(charter):` message |
| api→kanban notify adds coupling | Reuses the **existing** Redis lifecycle stream; no new transport |
| Runner's checkout lags the new commit | `writeRepoFile` commits to the same managed `base_path` checkout discovery reads; if runners use worktrees, add a pull before discovery read (verify during impl) |
| Losing agent-authored narrative nuance | Vision/Success Criteria become first-class memory categories — captured structurally, rendered into the doc; no expressiveness lost |
| Existing `CHARTER.md` files with hand-written Goals | First regeneration overwrites from DB — intended |

## 8. Out of Scope

- No new DB tables or migrations (categories are metadata strings).
- No feature flag — this is an evolution of always-on EPIC-203 functionality.
- Board / work-item linkage semantics for goals unchanged.
- The separate **Memory tab** (generic project memory) is untouched.
