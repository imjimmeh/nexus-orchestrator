# Workflow Catalog: Orchestration and Kanban

_Source reviewed: `seed/workflows/**` on branch `main` at commit `0ffc6d89`._

This catalog lists every seeded workflow currently present under `seed/workflows`. Event-triggered workflows are registered at API startup by `WorkflowEventTriggerService`, using trigger bindings resolved by `WorkflowTriggerRegistryService`. Manual/no-trigger workflows are launchable only through explicit launch paths or as child workflows.

## Trigger Evaluation Rules

- Event bindings are resolved from active workflow rows whose YAML `trigger.type` is `event`.
- Trigger name is `trigger.event` or `trigger.name`.
- `trigger.condition` is evaluated with Handlebars helpers `eq`, `and`, `or`, `not` against both root payload fields and `trigger.*`.
- Missing/blank condition means always trigger.
- Rendered condition must equal literal `true`; malformed templates evaluate false.
- Event payload `eventId` drives process-local event-trigger dedupe.
- Workflow `concurrency` drives active-run conflict behavior: `skip`, `queue`, or `cancel_running`.

## Kanban Status Automation Matrix

All status automation is based on `kanban.work_item.status_changed.v1` emitted by Kanban after a persisted status change.

| New status       | Workflow(s) expected                                                                                                                  | Should not run when                                                                                                                             |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| `refinement`     | `work_item_split_default` for large root items; `work_item_refinement_default` for normal/non-root cases.                             | Same-status updates; missing resource; large root item for refinement workflow; split workflow when item already has `metadata.split.parentId`. |
| `in-progress`    | `work_item_in_progress_default`.                                                                                                      | Item does not actually transition to `in-progress`; duplicate event ID already handled; workflow concurrency/dedupe skips.                      |
| `in-review`      | `work_item_in_review_default`.                                                                                                        | Status did not change or condition not literal true.                                                                                            |
| `ready-to-merge` | `work_item_ready_to_merge_default`.                                                                                                   | Status did not change or condition not literal true.                                                                                            |
| `todo`           | No direct status workflow. Eligible items may later be selected by dispatch (`work_item_todo_dispatch_default` or dispatch API/tool). | Dependencies incomplete, linked run exists, capacity full, branch claimed, dispatch limit reached.                                              |
| `blocked`        | No direct status workflow in current seeds.                                                                                           | CEO may inspect and decide next actions, but status itself does not trigger a workflow.                                                         |
| `done`           | No direct status workflow. Merge workflow emits `WorkItemMergeCompletedEvent`, which can trigger post-merge spec hydration.           | Automation suppressed status transitions do not emit follow-on Kanban status automation.                                                        |
